## All Bugs Fixed — Root Cause Summary

### Bug 1 — Duplicate pair causing double count (primary bug)
`detect_uds_pairs()` was returning **both** `(0x18DA3DF1, 0x18DAF13D)` **and** `(0x18DAF13D, 0x18DA3DF1)` — the same pair reversed. The reassembler ran twice over the identical frames, so 19 frames became 38 (or more). The dedup set used `payload[:8]` as key which was insufficient for multi-frame messages where the first 8 bytes are the same across blocks. **Fix:** dedup key is now `(round(ts, 4), bytes(full_payload))` — full payload hash — and `detect_uds_pairs()` now normalises every pair to `(min, max)` before inserting into the seen-set, so `(A,B)` and `(B,A)` are treated as the same pair.

### Bug 2 — Non-UDS SIDs passing through the decoder
After ISO-TP reassembly, `decode_uds_frames()` was accepting **any** payload, including Consecutive Frame sequences that came from non-UDS multi-frame traffic. If a J1939 periodic message happened to start a CF sequence (e.g. first byte `0x21`), it would be decoded as "UDS". **Fix:** Added an explicit SID whitelist check **before** calling `decode_uds_frames()`. Every reassembled payload is now validated:
- If `payload[0] == 0x7F` → the failing SID at `payload[1]` must be in `UDS_ALL_SIDS`
- Otherwise → `payload[0] & 0xBF` must be in `UDS_ALL_SIDS`
- Any payload that fails these checks is silently discarded — it is not a UDS frame

### Bug 3 — `self.CYAN` / `self.PURPLE` missing from PDF class
The new PDF sections (`_ecu_identity`, `_bus_load_analysis`) referenced `self.CYAN` and `self.PURPLE` but `PDFReportGenerator` only defined colours up to `TEAL`. **Fix:** Added `CYAN = colors.HexColor("#0097a7")` and `PURPLE = colors.HexColor("#7b1fa2")` to the class.
